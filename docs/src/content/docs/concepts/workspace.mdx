## Definition

A **Workspace** is Caracal's isolation boundary. It groups together:

- One active [Policy](/concepts/policy/) bundle (the OPA Rego rules that govern authority decisions).
- The [Principals](/concepts/principal/) and applications registered within it.
- The [Ledger](/concepts/ledger/) entries scoped to decisions made in this workspace.

Every mandate is issued within a workspace. The workspace determines which policy rules apply, and which principals are recognized. A principal from workspace A cannot receive a mandate governed by workspace B's policy.

Workspaces belong to a [Zone](/concepts/zone/). A zone can contain multiple workspaces. The zone owns the signing keys; the workspace owns the policies and principals.

## Why it exists

Large deployments need to isolate authority domains. A workspace for the `analytics-team` should have different policies than a workspace for the `billing-service`. Workspaces provide that isolation without requiring separate deployments.

Workspaces also enable multi-tenant SaaS patterns: each customer can have their own workspace with their own policies, their own application registrations, and their own audit history.

## Lifecycle

| Stage | Description |
|---|---|
| **Created** | Operator creates a workspace via `caracal init` or the coordinator API. A workspace is created inside an existing zone. |
| **Policy loaded** | Operator loads an OPA Rego bundle into the workspace. The STS caches the compiled bundle. |
| **Applications registered** | Applications are registered in the workspace with their `application_id` and authentication method. |
| **Active** | Principals in the workspace can request mandates; decisions are governed by the workspace's active policy. |
| **Policy updated** | New policy bundle loaded. The STS hot-reloads; in-flight requests complete against the previous bundle. |
| **Deleted** | Workspace deletion removes all registrations. Ledger data is retained according to the configured retention window. |

## Constraints and limits

- A principal belongs to exactly one workspace. Re-registering an application in a different workspace requires removing and re-adding it.
- A workspace belongs to exactly one zone. Cross-zone workspaces are not supported.
- Policy updates are atomic: the old bundle remains active until the new one is fully compiled and cached.
- Workspace deletion is permanent. There is no soft-delete or recovery window.
