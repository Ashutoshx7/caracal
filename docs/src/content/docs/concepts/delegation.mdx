## Definition

**Delegation** is the mechanism by which one [Principal](/concepts/principal/) transfers a scoped subset of its authority to another principal. The principal that delegates is the *delegator*; the one that receives authority is the *delegate*.

Delegation is managed by the [Coordinator](/architecture/services/). When a delegator calls the coordinator's delegation API, the coordinator records the delegation edge in its graph, issues a delegation token, and returns it to the delegator. The delegator passes this token to the sub-agent. The sub-agent presents the token to the coordinator to establish itself as a `delegate` principal.

## Why it exists

Multi-agent systems need controlled authority hand-off. An orchestrator agent that holds authority to call several APIs should be able to spin up specialized sub-agents and give each one only the authority it needs for its task — not a copy of the full authority.

Without delegation, the only option is to share credentials directly (which loses accountability) or re-authenticate each sub-agent independently (which loses the relationship between the orchestrator's authority and the sub-agent's authority).

Delegation maintains the chain: every mandate issued to a delegate is traceable back to the delegator's original mandate.

## The scoping invariant

**A delegate principal can never hold more authority than the delegator principal.**

This is enforced at the STS level. When the STS issues a mandate to a delegate principal, it intersects the requested intent with the delegator's current authority. If the delegator's mandate has a resource caveat for `/data/public/**`, the delegate cannot receive a mandate for `/data/sensitive/**` even if the workspace policy would otherwise permit it.

## Lifecycle

| Stage | Description |
|---|---|
| **Delegation issued** | Delegator calls coordinator `POST /sessions/{id}/delegate`. Coordinator records the delegation edge, issues a delegation token. |
| **Delegate established** | Sub-agent calls coordinator `POST /sessions/bind` with the delegation token. Coordinator establishes the sub-agent as a `delegate` principal scoped to the delegated authority. |
| **Active** | Delegate principal can request mandates within the delegated scope. |
| **Expired** | Delegation tokens have their own TTL (configured at issuance). After expiry, the delegate principal can no longer request new mandates via this delegation. In-flight mandates continue until their own expiry. |
| **Revoked** | Delegator or operator calls coordinator revocation. The `caracal.sessions.revoke` stream immediately invalidates all mandates held by the delegate. |

## Depth limits

Delegation chains have a configurable maximum depth (default: 3). An agent at depth 3 cannot further delegate. This prevents unbounded delegation trees that could obscure accountability.

## Example

```typescript title="Delegate authority to a sub-agent — TypeScript SDK"
import { CaracalClient } from '@caracalai/sdk'

// Orchestrator agent
const orchestrator = new CaracalClient({
  coordinatorUrl: process.env.CARACAL_COORDINATOR_URL,
  gatewayUrl: process.env.CARACAL_GATEWAY_URL,
})

const orchestratorSession = await orchestrator.run({
  intent: { action: 'file:read', resource: '/data/reports/**' },
})

// Delegate a narrower scope to the sub-agent.
// The sub-agent will only be able to read /data/reports/q1/*.
const delegation = await orchestratorSession.delegate({
  to: 'app:sub-agent-id',
  scope: {
    action: 'file:read',
    resource: '/data/reports/q1/*',
  },
  ttl: 120, // delegation token TTL in seconds
})

// Pass delegation.token to the sub-agent out-of-band.
// The sub-agent uses it to establish its delegate session:
//   const subAgent = await CaracalClient.fromDelegation({ delegationToken: delegation.token, ... })
```

## Constraints and limits

- Delegation requires the orchestrator to have an active session. You cannot pre-issue delegation tokens before establishing your own session.
- The delegated scope must be a subset of the delegator's current mandate scope. Requests to expand scope via delegation are rejected by the STS.
- Maximum delegation chain depth defaults to 3. Configurable in the coordinator's workspace settings.
- Delegation tokens cannot be renewed. A new delegation must be issued for extended operations.
