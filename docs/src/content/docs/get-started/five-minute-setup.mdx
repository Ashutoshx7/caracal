This path starts the released local stack, creates one protected resource from the Console, calls it through Gateway, and explains the audit evidence. Use a real upstream service that is reachable from the Gateway container or process.

## Prerequisites

- Docker 24+ with the Compose v2 plugin
- `caracal` and `caracal-console` installed from [Installation](../installation/)

## Start the stack

```bash
caracal up
caracal status --ready
```

`status --ready` must return success before you configure resources. If it fails, inspect `caracal status --ready --json` and the service logs from Docker Compose.

## Configure the first resource with Console

Open the Console:

```bash
caracal console
```

In the Console:

1. Choose **guided setup**.
2. Choose the selected zone, pick an existing zone, or type a new zone name.
3. Create a new agent app by typing its name, or press the picker key to select an existing app in the chosen zone.
4. Create a new protected resource by typing a plain resource name, or select an existing resource.
5. Enter only the Caracal scopes needed for the first successful call.
6. Confirm whether Console should create and activate the first access policy.
7. Optionally enter the upstream URL and first request path when you want Gateway routing and an exact generated Gateway check.
8. Optionally turn on **write files** when you want Console to create the runtime profile and secret file on this machine.
9. Review the setup plan, then create it.

Guided setup asks one question at a time and keeps internal IDs out of the main path. Leave **advanced** closed for the first run unless you need a credential provider, a custom resource identifier, custom profile paths, a custom token environment variable, or file overwrite behavior.

The result view shows which objects were created or selected, the one-time client secret for a newly created app, the access policy status, the active policy set, the generated runtime profile, local setup commands, file-write status, and first-success commands. If file writing was skipped, run the local setup commands, paste the revealed one-time client secret or selected app secret as the only line in the secret file, and keep the profile and secret files owner-readable only.

## Call the protected resource

Use the generated `CARACAL_CONFIG=... caracal run --` command prefix with the real command that starts your workload.

The token environment variable in the generated profile is derived from the resolved resource identifier. Use that injected token when your workload calls Gateway or the protected service. For Gateway-routed resources, the result view includes an exact `curl` command when first request path was provided; otherwise it shows the required Gateway URL and headers so you can run the request against a real protected path.

## Explain the first success

Return to the Console **audit** view after the first request. Select the request and open **explain** with its request ID. The explanation should show the created application, protected resource, Caracal scopes, policy version, policy set version, Gateway result, and final decision. If the request failed, use the same explanation to identify whether the next correction belongs in runtime config, policy, resource routing, or the upstream service.

## Validate the path repeatably

Use the repository validator when you need a CI-friendly first-success check for a real environment. The validator checks STS and Gateway readiness, sends a real mandate through Gateway to the resource path you provide, and confirms the Gateway success counter advanced. It does not create objects, mint tokens, or substitute a demo upstream.

```bash
CARACAL_TOKEN="$(cat ./runtime-token.txt)" \
node scripts/validateFirstSuccess.mjs \
  --gateway-url http://localhost:8081 \
  --sts-url http://localhost:8080 \
  --resource "$RESOURCE_ID" \
  --token-env CARACAL_TOKEN \
  --path /real/protected/path
```

Run it after Console guided setup or after your automation has created the zone, application, resource binding, active policy set, and short-lived mandate. Keep the token in your local shell or CI secret store; do not pass long-lived secrets in command history.

## Shut down

```bash
caracal down
```
