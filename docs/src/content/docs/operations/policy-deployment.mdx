## Rollout confirmation

Policy activation is not complete just because the API accepted the activation request. A safe rollout confirms three separate facts:

1. The database binding points at the intended policy set version.
2. The policy invalidation outbox row was dispatched to Redis.
3. STS has loaded the intended policy set version through its runtime policy engine.

After activating a policy set version, keep the returned `outbox_id` and poll the activation status endpoint from your automation or Console-backed workflow:

```bash
curl -fsS \
  -H "Authorization: Bearer $CARACAL_ADMIN_TOKEN" \
  "$CARACAL_API_URL/v1/zones/$ZONE_ID/policy-sets/$POLICY_SET_ID/activation-status?version_id=$POLICY_SET_VERSION_ID&outbox_id=$OUTBOX_ID"
```

The rollout is ready for protected traffic when:

| Field | Expected value |
| --- | --- |
| `active` | `true` |
| `outbox.state` | `dispatched` |
| `sts.state` | `loaded` |
| `sts.policy_set_version_id` | the activated policy set version |
| `propagation_status` | `loaded` |

If `propagation_status` is `waiting_for_outbox`, inspect `caracal_api_outbox_pending_total`, `caracal_api_outbox_oldest_pending_age_seconds`, Redis health, and API outbox logs. If it is `waiting_for_sts`, inspect STS readiness, `caracal_sts_opa_compile_errors_total`, `caracal_sts_opa_max_policy_age_seconds`, and the policy invalidation stream. If it is `failed`, stop the rollout and reconcile the outbox or policy contract before activating another version.

The STS status check confirms the STS service path reachable through `STS_URL`. In Kubernetes, keep the policy-age and compile-error alerts enabled so replica-level drift also becomes visible through metrics.

## Rollback confirmation

Rollback is the same workflow with the previous known-good policy set version:

1. Activate the previous version.
2. Poll `activation-status` with the rollback activation's `outbox_id`.
3. Run a policy simulation for the expected allow and deny cases.
4. Run one real Gateway request and confirm the request appears in Audit.

Do not close the rollback until activation status is `loaded` and the real Gateway request produces audit evidence with the expected policy set version.
