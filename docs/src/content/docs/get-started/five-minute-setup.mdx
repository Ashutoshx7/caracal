Start the local Caracal stack, register one public HTTP resource, call it through the Gateway, and inspect the audit record that proves the request passed through policy enforcement.

If a step fails, stop at that step and use the troubleshooting section before continuing.

## Prerequisites

- Caracal CLI installed — see [Installation](./installation/).
- Docker 24+ with the Compose v2 plugin.
- Outbound HTTPS access to `https://httpbin.org`.
- No existing `caracal.toml` in the working directory, or permission to overwrite it with `--force`.

## 1. Start the stack

```bash
caracal up
```

Verification:

```bash
caracal status --ready
```

The command must exit `0`. If it does not, run `caracal doctor --extended --json` and address the first failing dependency before continuing.

## 2. Protect one HTTP resource

```bash
caracal protect http \
  --identifier "resource://demo-api" \
  --upstream-url "https://httpbin.org" \
  --scopes "demo:read" \
  --user local-user \
  --write-config caracal.toml
```

This command creates or updates the zone, application, resource, policy, policy set, activation, grant, Gateway binding, and local `caracal.toml`.

The command writes `caracal.toml` with file mode `0600`. The `app_client_secret` in that file is shown once and should be handled as an application secret.

Verification:

```bash
caracal policy-set list
caracal resource list
```

Both must return at least one row. The active policy version id is the one shipped by `caracal protect http`, and the resource row should include `resource://demo-api`.

## 3. Verify token injection

```bash
caracal run -- sh -c 'printf "%s" "$RESOURCE_TOKEN" | caracal credential inspect -'
```

`caracal run` reads `caracal.toml`, exchanges the application client secret at the STS, and injects the returned bearer into the child process as `RESOURCE_TOKEN`.

Verification: the inspection output must include `verification=not_verified`, `zone`, `resource://demo-api`, and `demo:read`. The command decodes the token locally; it does not verify the signature.

## 4. Call the protected resource through the Gateway

```bash
caracal run -- sh -c 'curl -i -sS \
  -H "Authorization: Bearer $RESOURCE_TOKEN" \
  -H "X-Caracal-Resource: resource://demo-api" \
  "http://localhost:8081/get"'
```

Verification: the response headers include `HTTP/1.1 200 OK` or `HTTP/2 200`, and the JSON body includes `"url": "https://httpbin.org/get"`. Save the `X-Request-Id` response header if the call fails.

## 5. Confirm the audit trail

```bash
caracal audit tail --limit 5
```

The latest rows should include an `allow` decision for `resource://demo-api` and a Gateway action event. Pick the `request_id` for the Gateway call and run:

```bash
caracal explain <request_id>
```

The explanation reproduces the policy decision, the determining policies, and the upstream call result.

## Troubleshooting

### `refusing to overwrite caracal.toml`

`caracal protect http` refuses to replace an existing config file unless you opt in.

1. Move the existing file if you need to keep it:
   ```bash
   mv caracal.toml caracal.toml.backup
   ```
2. Re-run the `caracal protect http` command.

If you intend to replace the file, add `--force` to the command.

### `missing routing headers`

The Gateway did not receive `X-Caracal-Resource`.

1. Re-run the curl command exactly as shown in step 4.
2. Confirm the header value is `resource://demo-api`.

### `resource not configured`

The Gateway cache does not have a binding for `resource://demo-api`.

1. Confirm the resource exists:
   ```bash
   caracal resource list
   ```
2. Confirm readiness before retrying:
   ```bash
   caracal status --ready
   ```
3. If the resource was just created, wait up to 30 seconds for the Gateway binding cache to refresh, then retry the Gateway call.

### `upstream not addressable`

The Gateway rejected the upstream URL before proxying.

1. Confirm the resource upstream is `https://httpbin.org`.
2. If you changed the upstream to a private or loopback address, set the Gateway upstream policy intentionally. See [Gateway Behavior](/api/gateway/#upstream-ssrf-guard).

## What you have

You have:

- An active stack with service readiness checks green.
- One zone, one application, one resource, one active policy set.
- A working Gateway-mediated request to a protected HTTP upstream.
- Audit evidence and a `request_id` you can trace.
