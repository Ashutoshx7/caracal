## Definition

A **Zone** is a deployment unit that owns two classes of cryptographic material:

- **ES256 signing key pair** — the private key signs mandates; the public key is published as a JWKS endpoint. Every mandate's `zone_id` claim identifies which zone signed it, and the gateway uses the zone's JWKS to verify the signature.
- **Zone Encryption Key (ZEK)** — a symmetric key used to encrypt provider credentials stored in the STS vault. Provider credentials (OAuth tokens, API keys) are encrypted at rest per-zone; the ZEK decrypts them at exchange time.

A zone is the trust root for all signatures in a Caracal deployment. Gateways in a zone trust mandates signed by that zone's private key.

## Why it exists

Zones provide cryptographic isolation between deployment environments. A development zone uses development signing keys; a production zone uses separate production keys. A mandate signed by the development zone's key cannot be accepted by a gateway configured for the production zone.

Zones also enable multi-region or multi-tenant deployments where each region or tenant needs independent cryptographic authority.

## Relationship to workspaces

A zone contains one or more [Workspaces](/concepts/workspace/). Workspaces hold policies and principal registrations. The zone holds the keys that sign mandates for all workspaces within it.

The zone/workspace split separates *cryptographic authority* (zone) from *policy authority* (workspace). You can rotate zone keys without changing workspace policies, and you can update workspace policies without affecting the signing keys.

## JWKS publishing

The STS publishes the zone's public JWKS at:

```
GET /.well-known/jwks.json?zone_id=<zone_id>
```

Gateways cache this with a 5-minute TTL. On an unknown `kid` (key ID), the gateway forces a refresh against a 30-second rate limit.

## Key rotation

Zones support key rotation. When a new signing key is added to a zone:

1. The new key is added to the JWKS (both old and new keys are published).
2. New mandates are signed with the new key.
3. After the maximum mandate TTL elapses, all outstanding mandates signed by the old key have expired.
4. The old key is removed from the JWKS.

## Constraints and limits

- One STS instance manages one zone. To run multiple zones, deploy multiple STS instances.
- Gateways are zone-specific: a gateway is configured with one `zone_id` and only accepts mandates signed by that zone's keys.
- ZEK rotation is a destructive operation: all provider credentials must be re-encrypted with the new ZEK. Plan key rotation during a maintenance window.
- The `zone_id` claim in a mandate is verified by the gateway before any signature check. Cross-zone mandates are rejected before the signature is even validated.
